# Аудит загрузки данных, ресурсов и сериализации

## 1. Таблица рисков

| Участок MVP | Риск до усиления | Последствие | Защита после усиления |
| --- | --- | --- | --- |
| Входящие JSON body (`POST /api/auth/*`, `POST /api/payments`, `POST /api/fraud/*`) | Проверка была только по `Content-Length`; body с неизвестной длиной мог читаться Jackson до исчерпания входного потока | Переполнение буфера ввода, неконтролируемая загрузка данных в память, рост времени парсинга | Потоковое ограничение чтения тела запроса, ранний `413 Payload Too Large`, лимит документа и строк на уровне Jackson |
| Десериализация JSON | Не было ограничений на длину строк, глубину вложенности и размер чисел | Дорогой разбор deeply nested / oversized JSON, повышенная нагрузка на память и CPU | `StreamReadConstraints` для длины строки, глубины вложенности, длины числа и документа |
| Обработка файлов | В MVP нет бизнес-эндпоинтов загрузки файлов, но multipart-запрос мог дойти до MVC-слоя как неподдерживаемый тип | Неявное поведение, отсутствие централизованного контроля по файлам | Явное отклонение `multipart/form-data` в фильтре; в конфигурации добавлены лимиты файла/запроса как baseline |
| Постраничная загрузка данных из БД | Защита от больших страниц была только логикой `Math.min(...)`, но безопасная верхняя граница не была закреплена в config validation | Ошибочная конфигурация могла увеличить объём материализации результатов | Жёсткая валидация `defaultPageSize` / `maxPageSize` через `@Max(100)` |
| Ошибки парсинга и лимитов | Ошибки oversized/malformed payload возвращались слишком обобщённо | Непрозрачная диагностика и слабый контроль ошибок исполнения | Явные 4xx-ответы для oversized body и parser constraints, плюс журналирование причины |

## 2. Критичный сценарий

Сценарий: злоумышленник отправляет большой JSON в `POST /api/auth/register` или `POST /api/payments` без доверенного `Content-Length` либо с полем-строкой на тысячи символов.

До усиления:

- фильтр смотрел только на `Content-Length`;
- если длина не была объявлена или была обходной, Jackson продолжал читать поток и строить объект в памяти;
- ошибки парсинга и ограничения размера не были отделены от обычного `400`.

После усиления:

- тело читается через обёртку с подсчётом байт;
- превышение лимита прерывает чтение по контракту input stream;
- Jackson дополнительно режет слишком длинные строки и опасную вложенность;
- клиент получает детерминированный `413` или `400` без утечки деталей сервера.

## 3. До / После

| Поведение | До | После |
| --- | --- | --- |
| JSON > 16 KB | Проверялся только по `Content-Length`; при обходе заголовка мог читаться дальше | Всегда останавливается при достижении лимита и возвращает `413` |
| JSON-строка ~5 KB внутри допустимого body | Попадала в Jackson и дальше в валидацию DTO | Отклоняется на этапе парсера как небезопасный payload |
| Multipart-запрос в JSON API | Зависел от дальнейшего MVC-разбора | Отклоняется сразу на фильтре с предсказуемым ответом |
| Большие страницы history/flags через config | Риск неудачной конфигурации | Верхняя граница закреплена и валидируется при старте |

## 4. Ключевые изменения в коде

- `RequestSizeLimitFilter`: потоковое ограничение чтения body, контроль multipart и явное закрытие delegate stream.
- `JacksonSecurityConfig`: безопасные ограничения `StreamReadConstraints`.
- `GlobalExceptionHandler`: отдельные ответы для parser constraints и oversized payload.
- `AppProperties`: конфигурируемые лимиты запросов/файлов/JSON и безопасная валидация размеров страниц.
- Интеграционные тесты: oversized JSON, parser string limit и multipart rejection.
